# Deadsick

## Intro

This challenge pretends to be an "unhackable secure vault", but the real target is not a software binary. It is a hardware design written in Verilog.

The vault only starts leaking its contents when two different conditions are satisfied:

1. the key validator must accept the right input sequence
2. the VM core must emit its unlock signal

Only after both gates are open does the UART path start transmitting the ROM contents byte by byte.

So the goal is not to brute-force a flag string. The goal is to understand the control logic, trigger both unlock paths, and tap the internal byte stream.

## Mapping the Design

A quick module search gives the layout immediately:

```bash
$ rg -n "module\\s+vault_challenge_top|module\\s+key_validator|module\\s+vm_core" challenge_gate_final.v
742:module key_validator(clk, rst_n, key_byte, key_valid, key_ok, vm_enable);
869:module vault_challenge_top(clk, rst_n, data_in, data_valid, tx_pin);
1010:module vm_core(clk, rst_n, opcode, opcode_valid, unlocked);
```

And the lock controller appears as:

```bash
$ rg -n "vault_lock" challenge_gate_final.v | head
392:module \\$paramod\\vault_lock\\FLAG_BYTES=s32'00000000000000000000000001001110 (clk, rst_n, key_valid, vm_unlocked, uart_tx_done, window_open, tx_enable);
```

So the top-level structure is:

```text
data_in + data_valid
  -> key_validator  -> vm_enable
  -> vm_core        -> vm_unlock
  -> vault_lock     -> window_open + tx_enable
                          |
                          -> artifact_rom[addr] -> uart_tx -> tx_pin
```

The most important observation is this:

- we do not actually need to decode UART from `tx_pin`
- it is easier to tap the internal byte that is about to be transmitted

That is the hardware equivalent of reading a string before `printf()` sends it out.

Useful design details recovered from the RTL:

- the ROM address bus is `artifact_addr[6:0]`
- `FLAG_BYTES = 0x4e`, meaning 78 bytes are sent total
- in simulation, the first byte is just padding due to synchronous ROM timing
- the actual flag payload is therefore 77 bytes

```bash
$ timeout 2 ./vm_bytecode.out

```

So the practical decision is simple: ignore the noise and reverse the RTL directly.

## Unlock Condition 1: `key_validator`

The first gate is the `key_validator` module, which behaves like a small 2-bit state machine.

By tracing the state updates, the required four input bytes are:

```text
4B 45 59 F1
```

In ASCII, that is:

```text
"KEY" + 0xF1
```

Once those four bytes are accepted, the module raises `vm_enable`.

## Unlock Condition 2: `vm_core`

The second gate is the VM opcode stream.

The `vm_core` logic advances an internal 3-bit counter only when the opcode received is valid for the current state. The minimal working sequence recovered from the design was:

```text
1D 1E 1A 1B 00 00 00 00
```

The first four bytes are the meaningful prefix. The final four zeros are safe filler values that keep advancing the machine until the unlock pulse is emitted.

Once the eighth valid pulse lands, `vm_unlock` becomes active.

## Timing Detail: Why the First Byte Is Garbage

The `artifact_rom` is synchronous, which means data does not become valid in the same instant the address changes.

That creates one small simulation artifact:

- the first captured byte is stale or padding
- the real flag starts from the second observed transmission

So the clean solver simply skips the first `tx_fire` event.

## Solver Script

The cleanest solution is a Verilog testbench that:

1. generates a clock
2. pulses the correct key bytes
3. pulses the correct VM opcodes
4. watches the internal `tx_fire` signal
5. prints `dut.artifact_data` each time a byte is queued for UART transmission

I attached the archived solver as [solve_tb.v](./solve_tb.v). The full script is included below.

```verilog
`timescale 1ns/1ps
`default_nettype none

module solve_tb;
  // Simple clock generator.
  reg clk = 0;
  always #10 clk = ~clk;

  // DUT I/O.
  reg rst_n;
  reg  [7:0] data_in;
  reg        data_valid;
  wire       tx_pin;

  vault_challenge_top dut (
    .clk(clk),
    .rst_n(rst_n),
    .data_in(data_in),
    .data_valid(data_valid),
    .tx_pin(tx_pin)
  );

  // Captured bytes bookkeeping:
  // - `got` counts every tx_start pulse, including the first padding byte.
  // - `printed` counts only the actual printed flag bytes.
  integer got;
  integer printed;
  integer cycles;
  reg [7:0] out_bytes [0:255];

  // Edge detector for tx_start so we only capture once per transmitted byte.
  reg prev_tx_fire;

  // Drive one byte as a clean 1-cycle valid pulse.
  task automatic pulse_byte(input [7:0] b);
    begin
      @(negedge clk);
      data_in = b;
      data_valid = 1'b1;
      @(negedge clk);
      data_valid = 1'b0;
      data_in = 8'h00;
    end
  endtask

  initial begin
    // Reset and initialize.
    rst_n = 1'b0;
    data_in = 8'h00;
    data_valid = 1'b0;
    got = 0;
    printed = 0;
    cycles = 0;
    prev_tx_fire = 1'b0;

    repeat (5) @(posedge clk);
    rst_n = 1'b1;

    // Stage 1: unlock vm_enable via key_validator.
    pulse_byte(8'h4B); // K
    pulse_byte(8'h45); // E
    pulse_byte(8'h59); // Y
    pulse_byte(8'hF1); // final unlock byte

    // Stage 2: unlock the VM core.
    pulse_byte(8'h1D);
    pulse_byte(8'h1E);
    pulse_byte(8'h1A);
    pulse_byte(8'h1B);
    pulse_byte(8'h00);
    pulse_byte(8'h00);
    pulse_byte(8'h00);
    pulse_byte(8'h00);

    // One padding byte is emitted first, then the 77-byte flag payload.
    while (printed < 77 && cycles < 5_000_000) begin
      @(posedge clk);
    end

    if (printed < 77) begin
      $display("\n[!] Timeout: printed=%0d got=%0d vm_enable=%0d vm_unlock=%0d window_open=%0d tx_en=%0d byte_cnt=%0d",
               printed, got, dut.vm_enable, dut.vm_unlock, dut.window_open, dut.tx_en, dut.vault.byte_count);
    end else begin
      $write("\n");
    end
    $finish;
  end

  always @(posedge clk) begin
    cycles <= cycles + 1;

    prev_tx_fire <= dut.tx_fire;
    if (dut.tx_fire && !prev_tx_fire) begin
      // Each tx_fire corresponds to a new byte being queued into the UART.
      got = got + 1;
      if (got != 1) begin
        // Skip the first byte: it is a deliberate padding byte.
        out_bytes[printed] = dut.artifact_data;
        $write("%c", dut.artifact_data);
        printed = printed + 1;
      end
    end
  end

endmodule
```

## Pseudocode Summary

The control logic can be summarized like this:

```text
on reset:
  vm_enable   = 0
  vm_unlock   = 0
  window_open = 0
  tx_enable   = 0
  addr        = 0

feed bytes with data_valid pulses:
  if key_validator accepts 4 bytes:
    vm_enable = 1

  if vm_core sees 8 valid opcodes:
    vm_unlock = 1

if vm_enable && vm_unlock:
  window_open = 1
  tx_enable   = 1

while window_open && tx_enable:
  on uart_tx_done:
    addr += 1
    uart_tx_start(artifact_rom[addr])
```

And the recovered gate-level behavior becomes:

```text
if bytes_in == [0x4B, 0x45, 0x59, 0xF1]:
  vm_enable = 1

if opcodes[0:4] == [0x1D, 0x1E, 0x1A, 0x1B]:
  after 4 more valid filler bytes:
    vm_unlock = 1
```

## Closing

What makes this challenge good is that it teaches a classic hardware-reversing lesson:

- when a design protects output behind state machines and timing
- you do not necessarily need to decode the final serial line
- you can often tap the internal data path one stage earlier

That turns a "UART decoding" problem into a much cleaner "understand the unlock conditions" problem.
