# WRAPPED

## Intro

This challenge gives off a very specific energy immediately: one ZIP, one ELF, and a wrapper that exists only to unpack something else.

The joke string says:

> “wrap this in try and catch”

And that is basically the whole design.

The outer binary does three important things:

1. allocates an `RWX` memory region with `mmap`
2. inflates a zlib-compressed blob into that memory
3. calls the decompressed bytes as a function

So the real logic is not in the wrapper at all. The real target is the hidden second-stage payload.

The archive is very small. The only actual challenge binary is `wrapped`.

So the distribution is intentionally minimal. There is no helper library, no hidden config, no filesystem trick. Everything interesting is inside `wrapped`.

Running it shows the expected input style:

```bash

```

That tells us one small but important thing early:

- the program does not read from stdin
- it expects the candidate flag as `argv[1]`

So whatever hidden payload gets unpacked, it will most likely be a validation function that takes a string pointer as its argument.

## `main()`: `mmap`, `inflate`, and Call

Because the ELF is not stripped, the main symbols are easy to recover:

```bash
$ nm -an wrapped | grep -E ' main$| decompress_payload$| chall_z$| chall_z_len$'
0000000000001199 T decompress_payload
000000000000125a T main
0000000000004060 D chall_z
0000000000004c68 D chall_z_len
```

The disassembly of `main()` makes the wrapper behavior obvious:

```bash
$ objdump -d -Mintel --disassemble=main wrapped | sed -n '1,120p'

    126d: 74 0a                je     1279 <main+0x1f>
...
    128a: ba 07 00 00 00       mov    edx,0x7
    128f: be e0 29 00 00       mov    esi,0x29e0
    1299: e8 b2 fd ff ff       call   1050 <mmap@plt>
```

This is the high-level behavior:

- `mmap(..., prot=0x7)` allocates an `RWX` page
- the compressed blob `chall_z` is decompressed into that page
- the page is treated as code and called like a normal function
- the payload return value decides success or failure

So this is not a normal crackme where you reverse the outer ELF directly. The outer ELF is just a loader.

## `decompress_payload()` Is a Straight zlib Inflate

The decompression helper confirms that the second stage is zlib-compressed:

```bash
$ objdump -d -Mintel --disassemble=decompress_payload wrapped | sed -n '1,120p'
0000000000001199 <decompress_payload>:
...
    1216: e8 75 fe ff ff       call   1090 <inflateInit_@plt>
...
    1232: e8 f9 fd ff ff       call   1030 <inflate@plt>
...
    1241: e8 2a fe ff ff       call   1070 <inflateEnd@plt>
...
```

Conceptually, the function does:

1. build a `z_stream`
2. point it at the static compressed bytes in `.data`
3. point the output to the `RWX` mapping
4. call `inflate(..., Z_FINISH)`
5. require `Z_STREAM_END`

Once that is clear, the next step is obvious: extract the zlib blob without even running the code.

## Extracting the Payload

The ELF sections are enough to compute the exact file offsets:

```bash
$ readelf -S wrapped | grep -nE ' \\.(text|rodata|data) '
34:  [14] .text             PROGBITS         00000000000010b0  000010b0
38:  [16] .rodata           PROGBITS         0000000000002000  00002000
56:  [25] .data             PROGBITS         0000000000004040  00003040
```

From there:

- `.data` has runtime address `0x4040` and file offset `0x3040`
- `chall_z` is at runtime address `0x4060`, so the file offset is `0x3060`
- `chall_z_len` is at runtime address `0x4c68`, so the file offset is `0x3c68`

That gives a small Python extractor:

```bash
$ python3 - <<'PY'
import struct
import zlib

fn = "wrapped"

with open(fn, "rb") as f:
    f.seek(0x3c68)
    clen = struct.unpack("<I", f.read(4))[0]

    f.seek(0x3060)
    comp = f.read(clen)

print("chall_z_len =", clen)
print("zlib header =", comp[:2].hex())

out = zlib.decompress(comp)
print("decompressed =", len(out), "bytes")

open("payload.bin", "wb").write(out)
print("wrote payload.bin")
PY
chall_z_len = 3079
zlib header = 78da
decompressed = 10720 bytes
wrote payload.bin
```

## Reversing the Payload

The last-stage compare loop is the anchor that matters most. Using Rizin:

```bash
$ rizin -q -n -e scr.color=false -a x86 -b 64 -m 0 -c 'pd 30 @ 0x294d' payload.bin
            0x0000294d      lea   rax, [rsp + 0xb0]
            0x00002955      lea   rdx, [rsp + 0x70]
            0x0000295a      lea   rcx, [rsp + 0xe2]
...
        ┌─> 0x00002980      movzx ebx, byte [rdx]
        ╎   0x00002983      cmp   byte [rax], bl
       ┌──< 0x00002985      jne   0x29a0
       │╎   0x00002987      add   rax, 1
       │╎   0x0000298b      add   rdx, 1
       │╎   0x0000298f      cmp   rcx, rax
       │└─< 0x00002992      jne   0x2980
...
            0x0000299f      ret
...
            0x000029a7      mov   eax, 1
            0x000029ae      ret
```

This tells us the end condition very clearly:

- one buffer starts at `rsp + 0xb0`
- the other starts at `rsp + 0x70`
- the loop compares `0x32` bytes
- success returns `0`
- failure returns `1`

So all the obfuscation still reduces to the same structure as most reversing tasks:

```text
build expected[50] -> compare against user input -> return success or fail
```

The next question is how the expected 50-byte buffer is produced.

## The Actual Crypto Twist: “expand 67-byte k”

The first useful string anchors from `payload.bin` were:

```bash
$ strings -n 4 payload.bin | tail -n 8
)D$P
Proper MH
r Meltt
T$@H
t$ L
t$ H
T$pH
$91 /%awvl#85$a*A Renewal Agent CaptureTheFlag 7
```

Those strings are enough to reconstruct the crypto inputs:

- `A Renewal Agent CaptureTheFlag 7` looks like a 32-byte key phrase
- `Proper Meltt` is 12 bytes, which fits a ChaCha20 nonce
- `$91 /%awvl#85$a*` looks like an obfuscated constant

The payload uses a small byte transform to decode these values:

```bash
$ python3 - <<'PY'
def decode_byte(x: int) -> int:
    return ((x + 0x41) - (((x * 2) & 0x82))) & 0xff

s = b'$91 /%awvl#85$a*'
dec = bytes(decode_byte(b) for b in s)
print('obf :', s)
print('plain:', dec)
PY
obf : b'$91 /%awvl#85$a*'
plain: b'expand 67-byte k'
```

Normal ChaCha20 uses:

```text
expand 32-byte k
```

This payload changes it into:

```text
expand 67-byte k
```

The wording is modified, but the structure is still clearly ChaCha20-like:

- 16-byte sigma constant
- 32-byte key
- 12-byte nonce
- standard quarter rounds

Once the key, nonce, and sigma are reconstructed, decryption is straightforward.

## Solver Script

I attached the archived solver as [solve.py](./solve.py). The full script is included below.

```python
#!/usr/bin/env python3
"""
ARA7 - Wrapped (rev)

Reproduces the payload's ChaCha20-like decryption to recover the flag.

Notable twist:
  - sigma/constant is "expand 67-byte k" (decoded from an obfuscated 16-byte string)
  - key (32 bytes) and nonce (12 bytes) are derived by applying the payload's byte transform

Usage:
  python3 solve.py
"""

from __future__ import annotations

import struct


def decode_byte(x: int) -> int:
    """Payload's per-byte 'decode' transform."""
    # (x + 'A') - ((x*2) & 0x82)
    return ((x + 0x41) - (((x * 2) & 0x82))) & 0xFF


def decode_bytes(data: bytes) -> bytes:
    """Apply decode_byte for every byte."""
    return bytes(decode_byte(b) for b in data)


def rotl32(x: int, n: int) -> int:
    return ((x << n) & 0xFFFFFFFF) | (x >> (32 - n))


def quarterround(s: list[int], a: int, b: int, c: int, d: int) -> None:
    # Standard ChaCha quarter round
    s[a] = (s[a] + s[b]) & 0xFFFFFFFF
    s[d] ^= s[a]
    s[d] = rotl32(s[d], 16)

    s[c] = (s[c] + s[d]) & 0xFFFFFFFF
    s[b] ^= s[c]
    s[b] = rotl32(s[b], 12)

    s[a] = (s[a] + s[b]) & 0xFFFFFFFF
    s[d] ^= s[a]
    s[d] = rotl32(s[d], 8)

    s[c] = (s[c] + s[d]) & 0xFFFFFFFF
    s[b] ^= s[c]
    s[b] = rotl32(s[b], 7)


def chacha20_block(key32: bytes, counter: int, nonce12: bytes, sigma16: bytes) -> bytes:
    """
    Build one 64-byte ChaCha block.

    Layout (little-endian words):
        state[ 0.. 3] = sigma (16 bytes)
        state[ 4..11] = key   (32 bytes)
        state[12]     = counter
        state[13..15] = nonce (12 bytes)
    """
    if len(key32) != 32:
        raise ValueError("key must be 32 bytes")
    if len(nonce12) != 12:
        raise ValueError("nonce must be 12 bytes")
    if len(sigma16) != 16:
        raise ValueError("sigma must be 16 bytes")

    state = list(struct.unpack("<4I", sigma16) + struct.unpack("<8I", key32) + (counter,) + struct.unpack("<3I", nonce12))
    working = state.copy()

    # 20 rounds = 10 double-rounds
    for _ in range(10):
        # Column rounds
        quarterround(working, 0, 4, 8, 12)
        quarterround(working, 1, 5, 9, 13)
        quarterround(working, 2, 6, 10, 14)
        quarterround(working, 3, 7, 11, 15)

        # Diagonal rounds
        quarterround(working, 0, 5, 10, 15)
        quarterround(working, 1, 6, 11, 12)
        quarterround(working, 2, 7, 8, 13)
        quarterround(working, 3, 4, 9, 14)

    out = [(working[i] + state[i]) & 0xFFFFFFFF for i in range(16)]
    return struct.pack("<16I", *out)


def chacha_xor(data: bytes, key32: bytes, nonce12: bytes, sigma16: bytes, counter0: int = 0) -> bytes:
    """XOR data with generated keystream blocks."""
    out = bytearray()
    counter = counter0

    for i in range(0, len(data), 64):
        block = chacha20_block(key32, counter, nonce12, sigma16)
        counter = (counter + 1) & 0xFFFFFFFF
        chunk = data[i : i + 64]
        out.extend(a ^ b for a, b in zip(chunk, block))

    return bytes(out)


def main() -> None:
    # Anchors found in payload.bin via strings(1)
    sigma_obf = b"$91 /%awvl#85$a*"  # -> "expand 67-byte k"
    key_phrase = b"A Renewal Agent CaptureTheFlag 7"  # 32 bytes
    nonce_phrase = b"Proper Meltt"  # 12 bytes

    sigma = decode_bytes(sigma_obf)
    key = decode_bytes(key_phrase)
    nonce = decode_bytes(nonce_phrase)

    # Ciphertext constants recovered from the payload's stack immediates (little-endian)
    dwords = [
        0xA5B7C026,
        0x82831B9E,
        0xA10089A0,
        0x7034BA7D,
        0x5393AA9B,
        0x6246934F,
        0x225758FB,
        0xA8257589,
        0x732EC714,
        0x3213CE28,
        0x8DA3C239,
        0xCB9C8B3D,
    ]
    last_word = 0xB12D

    ciphertext = b"".join(struct.pack("<I", x) for x in dwords) + struct.pack("<H", last_word)

    plaintext = chacha_xor(ciphertext, key, nonce, sigma, counter0=0)
    print(plaintext.decode("ascii"))


if __name__ == "__main__":
    main()
```

## Pseudocode Summary

The wrapper itself is short and honest:

```text
main(argc, argv):
  if argc != 2:
    return 1

  buf = mmap(size=0x29e0, prot=RWX)
  if buf == MAP_FAILED:
    return 1

  if inflate(buf, chall_z, chall_z_len) != OK:
    return 1

  rc = ((int (*)(char *))buf)(argv[1])

  if rc == 0:
    puts("jolly good show m8")
  else:
    puts("get chinned m8")

  munmap(buf)
  return 0
```

And the payload boils down to:

```text
payload(user_input):
  expected[50] = decrypt(ciphertext, key, nonce, sigma="expand 67-byte k")
  return memcmp(user_input, expected, 50) ? 1 : 0
```

The compare loop offsets recovered from the payload were:

- compare loop start: `0x2980`
- success return `eax = 0`: `0x299b`
- fail return `eax = 1`: `0x29a7`

## Closing

This is a good challenge because it hides the real logic in a very familiar loader pattern:

- `mmap(RWX)`
- decompress a blob
- execute the blob as code

Once the second stage is extracted, the rest becomes much more structured than it first appears. The junk instructions are mostly noise. The real data flow is:

- decode constants
- derive key and nonce
- run a modified ChaCha20-like core
- compare 50 bytes

That makes this a good reminder that unpacking is often the real first step in reversing. Until the payload is isolated, all the wrapper analysis is just scaffolding.
